Ethics and Privacy Reflection

This study task management application was designed to support ethical data use and user privacy in simple ways. It only collects basic task information such as title, course details, due date, description, and notes, which helps avoid collecting unnecessary personal data.

  • Limited data collection: The app only asks for study task information needed for the main feature.
  • Input validation: Required fields are checked before a task can be saved, which helps keep user data accurate and reduces mistakes.
  • Restricted access: A login screen is used so task features are hidden until the user logs in.
  • Privacy support: Tasks are stored in the user's own browser using localStorage, so they are not sent to another user or shared publicly.

These choices show an effort to use data responsibly, keep the design simple, and protect user information within the limits of the Webpage Development assignment.

Ethical Decision-Making for the Login Design

The following applies the Markkula Center's ethical decision-making framework to the login system used in this application.

1. Recognize the Ethical Issue

The current login uses a hard-coded username and password written directly in JavaScript. While it limits access to the task page, anyone who views the source code can find the credentials. This raises a concern about whether this design adequately protects user privacy and security.

2. Get the Facts

  • The login credentials are stored in plain text inside a JavaScript file.
  • This application is an assignment project with no real server or database.
  • Tasks are saved in the user's own browser using localStorage, so no data is sent externally.
  • A more secure approach would require server-side authentication and proper password hashing.

3. Evaluate Alternative Actions using Ethical Lenses

  • Rights Lens — Fail: The current design does not fully protect the user's right to privacy because credentials are stored in plain text and can be found in the source code.
  • Justice Lens — Pass: All users are treated equally and given the same access under the same login conditions, which is fair.
  • Utilitarian Lens — Pass: For an assignment project, the simple login is practical and easy to understand, providing more benefit than harm in this limited context.
  • Common Good Lens — Fail: A hard-coded login does not benefit the broader community, as it sets a poor example for secure design in real applications.
  • Virtue Lens — Pass: Adding restricted access shows some responsibility and intention to limit who can use the task page, which reflects good development habits.
  • Care Ethics Lens — Fail: The design does not go far enough to protect users from security risks, showing a lack of full care for user safety.

4. Make a Decision and Test It

Overall, this login decision is partly ethical. A more ethical improvement would be to use a secure server, hashed passwords, and proper authentication so user login details are protected more responsibly.

5. Act and Reflect on the Outcome

For this assignment project, the hard-coded login is acceptable as a simple demonstration of access control. However, reflecting on this decision shows that real-world applications must prioritise secure authentication. If this project were developed further, the login system would need to be rebuilt with a proper server and secure password handling before it could be responsibly deployed to real users.
